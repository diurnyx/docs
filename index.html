Diurnyx Documentation

Note: Diurnyx is intended for research use only and is not a medical device. Recorded data is not suitable for diagnostic or clinical use.

Diurnyx is a research-oriented sensor data logger designed to record configured signals to EDF files on an SD card.

It is intended exclusively for experimental and scientific use. This guide describes typical operation, including preparing the card, starting and stopping a recording, and retrieving recorded data.

Overview

A recording session follows a basic workflow under normal operating conditions:

  1. Insert a prepared SD card.
  2. Power on the device.
  3. Short-press the button to start recording.
  4. Long-press the button to stop recording.
  5. Remove the SD card and copy the EDF file from the DATA directory.

Each session is written as a separate EDF file.

Requirements

  • A compatible SD card with a FAT filesystem
  • A valid config.json file in the root of the card
  • Connected sensors appropriate for the intended experimental setup
  • Device power

The device writes recordings under:

/DATA/

If needed, the device creates that directory automatically.

SD Card Format Overview

A prepared SD card is a standard SD card formatted and structured so the device can read it correctly. No special physical preparation is required beyond using a compatible card and the expected filesystem layout.

Short Version

A prepared SD card should have:

  • An MBR partition table
  • A FAT32 partition
  • A valid config.json file in the root directory

The DATA directory is optional before first use. If it is missing, the device creates it automatically when needed.

A typical card may look like:

SD_CARD/
├── config.json
└── DATA/
    └── 20260421_143015.edf

Format Details

The device expects the SD card to contain a standard partitioned FAT volume. In practical terms, this means:

  • The card should contain a Master Boot Record (MBR) at the beginning of the device.
  • The device searches for the first valid FAT32 partition.
  • The file config.json must be placed in the root of that FAT32 partition.

Cards formatted by common operating system tools as FAT are typically suitable, provided they use a normal MBR-based layout and not an unsupported filesystem such as exFAT or NTFS.

Expected Directory Layout

When viewed on a host computer, a correctly prepared SD card will usually appear similar to the following:

SD_CARD/
├── config.json
└── DATA/
    ├── 20260421_143015.edf
    ├── 20260421_150002.edf
    └── ...

Before the first recording, the card may also appear as:

SD_CARD/
└── config.json

Formatting Notes

  • Use a compatible SD card formatted with a supported FAT filesystem.
  • Ensure config.json is present in the root directory before inserting the card.
  • The DATA directory does not need to be created manually.
  • If unsure, reformat the card using standard FAT settings and then copy config.json to the root.

Controls

Idle

  • Short press: start a new recording
  • Long press: no action

Recording

  • Long press: stop the current recording
  • Short press: no action

Starting a Recording

  1. Insert the SD card.
  2. Power on the device.
  3. Wait for the device to become idle.
  4. Short-press the button.

When recording starts, the device attempts to read its configuration, initialize connected hardware, and create a new EDF file on the SD card.

Stopping a Recording

To stop a session, long-press the button while recording is active.

The device attempts to finalize and close the EDF file before returning to idle. Do not remove power or the SD card until the session has ended.

Recorded Files

Recorded files are stored in:

/DATA/

Each recording is stored as a separate EDF file. File naming is based on the recording start time as provided by the device clock:

YYYYMMDD_HHMMSS.edf

Example:

20260421_143015.edf

Timekeeping

The device uses its internal clock for timestamps, including file naming and metadata. Accuracy depends on prior time initialization and configuration.

If a timezone is configured, local time is used for the recording filename and metadata.

USB Logging

Logs may assist in observing device behavior during startup, card detection, recording start and stop, and error conditions.

Logs are intended for development and troubleshooting purposes and should not be interpreted as validated system diagnostics.

Troubleshooting

The following checks may help identify common issues during experimental use:

Recording does not start

  • Check that the SD card is inserted correctly.
  • Check that config.json exists in the root of the card.
  • Check that the card is readable and formatted as expected.
  • Check that the connected hardware matches the intended setup.
  • Review USB logs if available.

No useful data is recorded

  • Check sensor connections.
  • Check that the intended setup is installed.
  • Review USB logs for hardware or acquisition errors.

Timestamps look wrong

  • Check device time initialization.
  • Check the configured timezone, if used.
